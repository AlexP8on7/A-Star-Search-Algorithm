What is the A* Algorithm?

Before diving into the implementation, it is important to understand what the A* algorithm actually does. A* is a pathfinding algorithm that finds the shortest path between two points on a grid. It does this by evaluating each position using the formula f(n) = g(n) + h(n), where g(n) is the cost from the start node to the current node, and h(n) is the heuristic estimate of the cost from the current node to the goal. By combining these two values, A* always explores the most promising path first, making it both efficient and accurate.

Implementation

The implementation was structured across multiple files to ensure modularity and reusability. The core algorithm lives in aStarPathfinding.cpp, with declarations in aStarPathfinding.h, keeping the interface clean and separate from the logic. main.cpp remains minimal, simply calling RunAllTests() from PathfindingTests.cpp, which validates the algorithm across three scenarios: a simple path with no obstacles, a path that must navigate around obstacles, and a case where no path exists.

Modern C++ features were applied throughout the implementation. Smart pointers (std::shared_ptr) replaced raw pointers to eliminate memory leaks and ensure safe resource ownership[1]. The STL containers std::vector, std::priority_queue, and std::unordered_set were used for the grid representation, open set, and closed set, respectively. Lambda functions enabled efficient O(1) hashing of grid positions for closed-set lookups, while range-based for loops with auto improved code readability and type safety[1]. 

// Lambda to convert (x,y) to a unique hash for O(1) lookup
auto hashPos = [](int x, int y) { return x * 10000 + y; };

// Manhattan distance heuristic
float ManhattanHeuristic(std::shared_ptr<Node> a, std::shared_ptr<Node> b) {
    return static_cast<float>(std::abs(a->x - b->x) + std::abs(a->y - b->y));
}

Design Choices

The Manhattan heuristic was selected because it is ideally suited for grid-based movement restricted to four directions (up, down, left, right). It estimates the remaining distance as the sum of the absolute horizontal and vertical differences between nodes, which matches the movement model in the implementation.

The grid is represented using integer values: 0 denotes an empty (traversable) cell, 1 indicates an impassable obstacle, and values greater than 1 represent weighted terrain with increased traversal cost. This design allows the A* algorithm to naturally handle weighted graphs without requiring any modifications to the core logic.

For the closed set, an std::unordered_set of hashed grid positions was chosen instead of a std::vector, providing average O(1) lookup time rather than O(n). This specific design choice significantly improves performance on larger grids, while still complying with guidelines that mandate deterministic behavior and exception-safe usage of standard containers in safety-related systems[1].

Project Management

Development was carried out consistently across lab sessions, with each session building on the previous. Claude Sonnet was used to help explain certain aspects of the A* algorithm, particularly understanding how f(n) = g(n) + h(n) drives node selection. Visual Studio 2022 was used throughout for building and debugging. Progress was demonstrated during lab checkpoints, with each session introducing improvements such as migrating from raw pointers to smart pointers, fixing the closedSet bug where nodes were being revisited, and separating test logic from the main entry point.

The UML diagram above shows how the architecture shifted from a monolithic design to a modular one. The old structure had all logic crammed into main.cpp, while the refactored version separates responsibilities across dedicated files.

An early version of main.cpp contained some of the pathfinding logic directly, including grid setup, obstacle placement, and output. This was replaced as it violated the principle of modular design. main.cpp should act as an entry point, not contain implementation details. Refactoring the code into separate components improved maintainability, readability, and testability, making it easier to validate individual parts of the system. This emphasis on structured, testable code became even more significant when considering real-world software failures such as CrowdStrike Global IT outage, where a faulty update caused widespread system crashes across critical infrastructure globally[3]. It highlights how insufficient validation or tightly coupled systems can lead to large-scale issues. As a result, this project placed greater importance on incremental development, modular design, and thorough testing before integrating changes. Reflection

The biggest challenge was developing a solid understanding of how the A* algorithm functions, particularly how the evaluation function f(n) = g(n) + h(n) directs the search toward the most promising paths. Once it became clear that g(n) represents the exact cost came from the start to the current node, while h(n) provides an actual estimate of the remaining cost to the goal, the role of the priority queue began to make sense. It always expands the node with the lowest total estimated cost first.

In this project, modern C++ features such as smart pointers, lambda expressions, and STL containers were adopted to improve code quality. This approach contributed to safer memory management and more maintainable code, particularly when compared to other techniques. Reflecting on this decision, it became clear that these features not only reduced the likelihood of common errors, such as memory leaks, but also aligned the implementation more closely with established guidelines for developing reliable software in critical and safety-related systems[1]. However, their effective use required a deeper understanding of modern C++ implementations, highlighting the importance of balancing suitable language features with code readability and team familiarity.
