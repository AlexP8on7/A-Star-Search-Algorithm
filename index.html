What is the A Star Algorithm?

Before diving into the implementation, it is important to understand what the A Star algorithm actually does. A Star is a pathfinding algorithm that finds the shortest path between two points on a grid. It does this by evaluating each position using the formula f(n) = g(n) + h(n), where g(n) is the cost from the start node to the current node, and h(n) is the heuristic estimate of the cost from the current node to the goal. By combining these two values, A Star always explores the most promising path first, making it both efficient and accurate.

Implementation

The implementation was structured across multiple files to ensure modularity and reusability. The core algorithm lives in aStarPathfinding.cpp, with declarations in aStarPathfinding.h, keeping the interface clean and separate from the logic. main.cpp remains minimal, simply calling RunAllTests() from PathfindingTests.cpp, which validates the algorithm across three scenarios: a simple path with no obstacles, a path that must navigate around obstacles, and a case where no path exists.

Modern C++ features were applied throughout. Smart pointers (std::shared_ptr) were used in place of raw pointers to eliminate memory leaks. The STL containers std::vector, std::priority_queue, and std::unordered_set were used for the grid, open set, and closed set respectively. Lambda functions were used to hash grid positions for O(1) closed set lookup, and range-based loops with auto were used throughout for readability.

Design Choices

The Manhattan heuristic was chosen as it is well suited for grid-based movement where only four directions are permitted. It calculates distance as the sum of horizontal and vertical steps, which directly reflects how movement works in this implementation. The grid itself uses integer values, where 0 represents an empty cell and 1 represents an obstacle. Cells with values greater than 1 are treated as weighted terrain, meaning the algorithm naturally supports weighted graphs without any additional changes.

The closed set uses an std::unordered_set of hashed positions rather than a vector, giving O(1) lookup instead of O(n). This was a deliberate design decision to improve performance on larger grids.

Testing & Validation

Three test cases were written to cover both general and edge scenarios. Test 1 validates that the algorithm finds a path on an open grid. Test 2 confirms the algorithm correctly navigates around obstacles. Test 3 verifies the algorithm returns an empty path when the goal is completely unreachable, with a wall spanning the entire grid. Output is visualised using S for start, G for goal, * for the path, # for obstacles, and . for empty cells.

Reflection

The biggest challenge encountered was wrapping my head around how the A* algorithm actually works, specifically how f(n) = g(n) + h(n) guides the search. Once I understood that g(n) represents the known cost and h(n) represents the estimated remaining cost, the logic of the priority queue became much clearer — the algorithm always processes the node with the lowest total cost first.

One thing I would do differently given more time is unify the visual output across all test cases. Currently TestNoPath calls PrintGrid while TestSimplePath and TestWithObstacles call PrintGridWithPath, resulting in an inconsistent output style. The fix would be straightforward — having PrintGridWithPath fall back to PrintGrid when the path is empty — but time did not allow for this.

The current implementation is limited to four directional movement, meaning diagonal paths are not possible. This is a known limitation of the Manhattan heuristic, which assumes only horizontal and vertical movement. Given more time, diagonal movement would be the next feature to implement, which would require switching to the Chebyshev or Octile distance heuristic and expanding the direction vectors from 4 to 8.
